# DB와 DBMS 개념 및 CRUD 개요
### 1. DB (Database, 데이터 베이스)
- 데이터의 집합 형식을 갖추지 않아도 엑셀 파일을 모아 둔다면 그것 또한 데이터베이스임 

---

### 2️. DBMS (Database Management System, 데이터베이스 관리 시스템)
- 데이터를 효과적으로 관리하기 위한 시스템 
- 개인이 파일을 여러 개 묶어서 폴더에 보관하면 데이터를 찾고 관리하는데 많은 비용이 발생 
- 이를 보다 시스템적으로 작동하게 만든 시스템 
- 예) ORACLE, MYSQL 등 

---

### 2-1. 관계형 데이터베이스 구성 요소 
- **테이블**
   - DBMS의 DB 안에서 데이터가 저장되느 형식 
- **스키마**
   - 테이블이 어떠한 구성으로 되어있는지, 어떠한 정보를 가지고 있는지에 대한 기본적인 구조를 정의 
   
---
   
### 3. CRUD 개요
**CRUD**는 데이터베이스에서 데이터를 처리하는 기본적인 네 가지 작업
1. **Create (생성)**: 새로운 데이터 데이터베이스에 삽입
2. **Read (읽기)**: 데이터베이스에서 데이터 조회
3. **Update (수정)**: 기존 데이터 수정
4. **Delete (삭제)**: 데이터 삭제

---

### 4. 테이블 구조 

| 직원ID | 이름  | 나이 | 부서   |
|--------|-------|------|--------|
| 1      | 지민  | 25   | 개발팀 |
| 2      | 철수  | 30   | 디자인팀 |

---

#### 4-1. 행(Row)
- 테이블에서 **가로**로 구성된 데이터 단위 
- 하나의 행은 **하나의 엔터티에 대한 정보**를 포함 
   - 예) **직원 정보**를 저장하는 테이블이 있다면, **하나의 행은 하나의 직원에 대한 모든 정보**를 나타냄 

- 직원 테이블에서 '직원 1'의 정보를 나타내는 하나의 행은 직원의 이름, 나이, 부서 등의 데이터를 포함

- 위 테이블에서 첫 번째 행은 '지민'에 대한 정보를 포함하고 있다.

---
#### 4-2. 열(Column)
- 테이블에서 **세로**로 구성된 데이터 단위 
- 각 열은 **특정 속성**이나 **특징**을 나타냄 

- 예) 직원 테이블에서 **이름**, **나이**, **부서**는 각각 하나의 열 
- 위 테이블에서 **이름**, **나이**, **부서**가 각각 열로 존재하고, 이 열들에 맞춰 데이터가 저장 

---
#### 4-3. 튜플(Tuple) & 레코드(Record)
✅ **튜플**
- 행(Row)과 동일한 개념으로 사용되는 용어 
- **하나의 데이터 항목 == 하나의 행 **

- **여러 개의 값을 가지며**, 각 값은 테이블의 열에 해당하는 데이터이다. 
- 위 테이블에서 **"1, 지민, 25, 개발팀"**은 하나의 **튜플**

✅ **레코드**
- 데이터베이스에서 **데이터를 표현하는 기본적인 단위**로

- **튜플과 같은 의미**로 사용

---

### 정리 
- **행(Row)**: 데이터베이스 테이블에서 가로로 구성된 데이터 단위.
- **열(Column)**: 테이블에서 세로로 구성된 데이터 단위, 각 열은 특정 속성에 해당.
- **튜플(Tuple)**: 하나의 행, 즉 하나의 데이터 항목.
- **레코드(Record)**: 튜플과 동일한 의미로 사용되며, 하나의 데이터 항목.

---

| 직원ID | 이름  | 나이 | 부서   |
|--------|-------|------|--------|
| 1      | 지민  | 25   | 개발팀 |
| 2      | 철수  | 30   | 디자인팀 |

- **행/튜플/레코드**: 
    - "1, 지민, 25, 개발팀" → 하나의 행(튜플/레코드)
    - "2, 철수, 30, 디자인팀" → 또 다른 행(튜플/레코드)
  
- **열**: 직원ID, 이름, 나이, 부서 → 각각의 속성

- **스키마**: 직원 테이블의 스키마는 아래와 같이 정의
  - 직원ID (정수형)
  - 이름 (문자형)
  - 나이 (정수형)
  - 부서 (문자형)

### 5. 스키마 설계 개념
#### 데이터베이스의 구조와 규칙을 정의한 설계도
#### 사용자의 관점과 실제 설계된 물리적인 방식을 분리하기 위해 고안 

---

#### 5-1. 외부 스키마 
- **사용자가 보는 관점에서** 데이터베이스 스키마를 정의 
- 사용자나 응용 프로그램이 필요한 데이터를 정의 

#### 5-2. 개념 스키마 
- 사용자 관점의 데이터베이스 스키마를 통합하여 **데이터베이스의 전체 논리적 구조를 정의**
- 전체 데이터베이스의 개체, 속성, 관계, 데이터 타입 등을 정의 

#### 5-3. 내부 스키마 
- 데이터가 **물리적으로 어떻게 저장되는지를 정의** 
- 데이터의 저장 구조, 컬럼, 인덱스 등을 정의함 

> #### 3단계 스키마의 독립성 
- **독립성 **
   - 물리적, 논리적 구조를 변경하더라도 사용자가 사용하는 응용 프로그램에 영향을 주지 말아야 함 
---
1) **논리적 독립성** 
논리적 데이터 구조가 변경되어도(개념 스키마 변경) 응용 프로그램에 영향을 주지 않는 특성 
2) **물리적 독립성**
물리적 구조가 변경되어도(내부 스키마 변경) 개념/외부 스키마에 영향을 주지 않는 특성
